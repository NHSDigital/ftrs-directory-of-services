# ETL ODS Performance Testing

Scripts for load testing and monitoring the ETL ODS pipeline (Extractor → Transformer → Consumer).

## Testing Approaches

There are two approaches for performance testing the ETL ODS pipeline:

| Approach | What it tests | How it works |
|----------|---------------|--------------|
| **1. Trigger Extractor** | Full pipeline (Extractor → Transformer → Consumer) | Invokes the Extractor Lambda with a specific date, which calls the real ODS API and pushes results through the full pipeline |
| **2. SQS Injection** | Transformer → Consumer only | Injects synthetic FHIR Organisation messages directly onto the SQS transform queue, bypassing the Extractor |

Both approaches use `make etl-metrics` to capture pipeline latency from CloudWatch.

## Prerequisites

- AWS CLI configured with valid credentials/SSO
- Python 3.11+ with `boto3`
- Access to the target AWS environment
- Run `extract_organisation_ids.sh` to extract real ODS codes from DynamoDB before injecting SQS messages:

  ```sh
  make ds-data-prep ENVIRONMENT=dev
  ```

  This generates `parameter_files/crud_organisation_ids.csv` used by the SQS injection script.

## Approach 1: Trigger Extractor Lambda

Invoke the Extractor Lambda directly with a specific date. It calls the real ODS API, fetches organisations modified on that date, and pushes them through the full pipeline. The date must be in `YYYY-MM-DD` format and no more than 185 days in the past.

### Quick Start

```sh
cd tests/performance

# Trigger extractor for a specific date
make etl-trigger-extractor ENVIRONMENT=dev DATE=2026-03-09 WORKSPACE=default

# Trigger on a non-default workspace
make etl-trigger-extractor ENVIRONMENT=dev DATE=2026-03-09 WORKSPACE=my-branch

# Then capture latency
make etl-metrics ENVIRONMENT=dev
```

### Parameters

| Variable      | Description                             | Default      |
|---------------|-----------------------------------------|--------------|
| `ENVIRONMENT` | Target AWS environment                  | (required)   |
| `DATE`        | Date to extract (YYYY-MM-DD)            | (required)   |
| `WORKSPACE`   | Terraform workspace                     | `default`    |
| `AWS_REGION`  | AWS region                              | `eu-west-2`  |

### How It Works

- Invokes `ftrs-dos-{env}-etl-ods-extractor-lambda{-workspace}` with a `{"date": "YYYY-MM-DD"}` payload.
- The Extractor calls the ODS API for organisations modified on that date.
- Results are pushed onto the transform queue and flow through the full pipeline.
- Volume depends on how many organisations were actually modified on the given date.

## Approach 2: SQS Injection

Inject synthetic FHIR Organisation messages directly onto the SQS transform queue, bypassing the Extractor Lambda. This allows testing the Transform and Load stages at volumes beyond what the real ODS API provides.

### Quick Start

```sh
cd tests/performance

# Generate real ODS codes first (recommended)
make ds-data-prep ENVIRONMENT=dev

# Dry run — preview messages without sending
make etl-inject-sqs-dry-run ENVIRONMENT=dev COUNT=5

# Inject messages
make etl-inject-sqs ENVIRONMENT=dev COUNT=1000

# Inject to a non-default workspace
make etl-inject-sqs ENVIRONMENT=dev WORKSPACE=my-branch COUNT=500
```

Or run the script directly:

```sh
ENVIRONMENT=dev COUNT=100 python scripts/etl_ods/generate_synthetic_sqs_messages.py
```

### Parameters

| Variable      | Description                             | Default      |
|---------------|-----------------------------------------|--------------|
| `ENVIRONMENT` | Target AWS environment                  | (required)   |
| `WORKSPACE`   | Terraform workspace                     | `default`    |
| `COUNT`       | Number of messages to generate and send | `100`        |
| `BATCH_SIZE`  | SQS batch size (max 10)                 | `10`         |
| `AWS_REGION`  | AWS region                              | `eu-west-2`  |
| `DRY_RUN`     | Preview messages without sending        | `false`      |

### How It Works

- If `parameter_files/crud_organisation_ids.csv` exists (generated by `make ds-data-prep`), the script uses real ODS codes from DynamoDB, cycling through rows when `COUNT` exceeds the number of rows.
- Otherwise it falls back to synthetic `PERFxxxx` codes.
- Each message is wrapped in the envelope the Transformer expects: `{"organisation": <FHIR resource>, "correlation_id": "...", "request_id": "..."}`.
- Messages are sent to `ftrs-dos-{env}-etl-ods-transform-queue{-workspace}` (workspace suffix omitted for `default`).

## CloudWatch Metrics

Fetch Lambda latency percentiles, SQS queue health, DLQ depth, and EventBridge schedule metrics for the ETL ODS pipeline.

### Quick Start

```sh
cd tests/performance

# Console output (last 1 hour)
make etl-metrics ENVIRONMENT=dev

# Console + CSV export
make etl-metrics-csv ENVIRONMENT=dev

# Custom hours window
make etl-metrics ENVIRONMENT=dev HOURS=4

# Explicit start/end times
make etl-metrics ENVIRONMENT=dev START='2026-03-10 14:00' END='2026-03-10 15:30'

# Start only (end defaults to now)
make etl-metrics-csv ENVIRONMENT=dev START='2026-03-10 14:00'

# Non-default workspace
make etl-metrics ENVIRONMENT=dev WORKSPACE=my-branch
```

### Parameters

| Variable      | Description                                 | Default      |
|---------------|---------------------------------------------|--------------|
| `ENVIRONMENT` | Target AWS environment                      | (required)   |
| `WORKSPACE`   | Terraform workspace                         | `default`    |
| `HOURS`       | Hours to look back                          | `1`          |
| `START`       | Explicit start time (e.g. `2026-03-10 14:00`) | —          |
| `END`         | Explicit end time (default: now)            | —            |
| `AWS_REGION`  | AWS region                                  | `eu-west-2`  |
| `AWS_PROFILE` | AWS named profile (optional)                | —            |

### CLI Flags

| Flag             | Description                                          |
|------------------|------------------------------------------------------|
| `--hours`, `-t`  | Hours to look back (default: 1)                      |
| `--start`        | Explicit start time (e.g. `'2026-03-10 14:00'`)      |
| `--end`          | Explicit end time (default: now). Use with `--start`. |
| `--profile`      | AWS profile name                                     |
| `--export-json`  | Export full metrics to JSON                          |
| `--export-csv`   | Export latency summary to CSV                        |

When `--start` (and optionally `--end`) is provided it takes precedence over `--hours`. This applies to both Make targets and direct script usage.

### Output

**Console** — Per-resource breakdown with duration percentiles (p50/p90/p95/p99), invocations, errors, throttles, concurrent executions, queue depths, and a summary table.

**CSV** (`reports/etl_latency_report_{timestamp}.csv`) — One row per resource with columns: Resource, P50, P90, P95, P99, Avg, Max, Invocations, Errors.

**JSON** (`reports/etl_metrics_{timestamp}.json`) — Full time-series data for all resources.

### Resources Monitored

| Category     | Resources                                                                |
|--------------|--------------------------------------------------------------------------|
| Lambda       | `extractor-lambda`, `transformer-lambda`, `consumer-lambda`              |
| SQS Queues   | `transform-queue`, `load-queue`                                          |
| SQS DLQs     | `transform-dlq`, `load-dlq`                                             |
| EventBridge  | `ods-etl-schedule`                                                       |

All resource names are prefixed with `ftrs-dos-{env}-etl-ods` and suffixed with `-{workspace}` (omitted for `default`).
